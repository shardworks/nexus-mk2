# X020 Findings — Code-Lookup for the Implementer

**Verdict:** H1 cost-target **not supported** on N=1 (−20.6% < 25% threshold). H1 mechanism **didn't fire** on N=1 (0% adoption vs ≥20% target) — but the transcript walk shows the implementer *had* roughly 6 eligible find-callers/find-definition queries and chose Grep for all of them. The failure isn't "the role doesn't ask the question"; it's "the role asks the question and bypasses the structured tool anyway." Different diagnosis than X019, different implication for follow-up.

## Hypothesis

Providing the **implementer** (Spider's `implement` engine, role `artificer`) with the X019 `code-lookup` MCP tool — modes `symbol`, `usages`, `package`, backed by the precomputed reverse-usage-index artifact — reduces implement-session cost ≥25% relative to a clean baseline. Tier 1 / Tier 2 spec quality must not regress. Adoption-metric target: ≥20% of tool calls reach for `code-lookup` (one order of magnitude above X019's 3% planning-side rate).

X020 ports the X019 hypothesis to a different pipeline phase. X019's diagnosis was role mismatch: the planner reads deeply, doesn't ask "find all callers" questions. The implementer's edit workflow ("change `setWritExt`'s signature" → "find every caller") is canonically aligned with `code-lookup`'s `usages` mode. If the diagnosis was correct, the implementer should produce the cost-reduction signal X019 didn't.

## Diagnosis

**The simplest version of H1's mechanism — "the model invokes code-lookup, code-lookup queries are cheaper than Grep+Read, therefore cost drops" — did not fire.** The −20.6% cost gap with 0% adoption rules it out at N=1.

What might explain the cost gap (with the tool unused)?

1. **N=1 session-to-session variance.** Implementer behavior on this brief may have ≥20% noise floor. We have no paired re-runs to estimate it.
2. **Partial scope shift.** With-tool produced 3 fewer doc files. At a few percent of the gap; doesn't fully explain it.
3. **Snippet behavioral side-effect.** The snippet was added as a top-level "Tool preference" section in the artificer role file. Its framing ("structural queries via code-lookup, textual via Grep") may have nudged the model toward more focused tool sequences without invoking code-lookup. The 25% drop in total tool calls (155 → 116) and 52% drop in Edit calls (44 → 21) are compatible with a more deliberate session shape.
4. **Implementation-attempt artifacts.** The first attempt failed mid-flow (80 tool calls of progress, then $0/0-token crash). Retry succeeded. Possible the second attempt benefited from some implicit state effect not visible in the data.

**Crucially, X020 weakens — not supports — the X019 role-mismatch diagnosis** *in a different way than first thought*. The initial read was: "the implementer's workflow on this brief didn't generate find-callers queries." A walk of the with-tool transcript falsifies that read. Of 11 Grep calls, **roughly 6 were canonical code-lookup workflows**:

| Grep | Pattern | code-lookup equivalent |
|---|---|---|
| #3 | `runConformanceSuite\|SqliteBackend` | `mode=symbol name=SqliteBackend` |
| #4 | `SqliteBackend` | `mode=usages name=SqliteBackend` (textbook) |
| #5 | `new SqliteBackend\|SqliteBackend\(` | `mode=usages name=SqliteBackend` (returns `instantiation` ref-kind) |
| #7 | `dropBook` | `mode=usages name=dropBook` |
| #9 | `interface Apparatus\|ApparatusDefinition` | `mode=symbol name=Apparatus` |
| Bash-grep | `createCartograph\|\.start(` in test | `mode=usages name=createCartograph` |

The implementer **did** ask find-callers and find-definition questions — about half its Greps were exactly the "where is this exported symbol used?" or "how is this exported interface declared?" shape that code-lookup is designed to answer, and that the snippet in `roles/artificer.md` explicitly called out with worked examples. **It just didn't use code-lookup to answer them.**

This is a different failure mode than X019. X019's read was "the role doesn't ask these questions." X020's evidence is "the role asks these questions and doesn't reach for the tool designed to answer them, even when explicitly prompted." Possible explanations:

1. **Strong training prior on Grep.** Claude has heavy training-time exposure to Grep as *the* search tool. A single role-file section doesn't displace that prior in tool-selection — especially since Grep was already in the model's hands and code-lookup was new.
2. **Snippet positioning effect.** The snippet sits in `roles/artificer.md` which loom appends to the system prompt. It may be too far (in token-distance) from the per-turn tool-selection signal to influence decision-making at the turn level.
3. **Snippet phrasing weakness.** "Reach for code-lookup aggressively" + worked examples didn't outweigh defaults. Stronger imperative-with-prohibition phrasing ("Do not use Grep when you have an exported symbol name") might shift adoption — or might fail the same way.
4. **Tool surface novelty.** The model may need first-hand observation of code-lookup's output shape before treating it as a real option. With zero invocations, the feedback loop never engaged.

The dropBook brief was a downstream-application brief (adding a primitive, applying it once), not a refactoring brief — but the implementer's audit work *was* refactoring-shaped (verify all SqliteBackend instantiation sites, find all ChangeEvent consumers, locate the Apparatus interface). The find-callers shape was there. The tool-selection wasn't.

## Decision

**The on-demand-cross-reference-tool line is not dead, but it's clearly not a one-snippet-and-done lever.** The diagnostic shifts:

- **X019 + X020 do NOT together rule out the mechanism.** The transcript walk above shows the implementer *had* the queries; the question becomes whether the model can be persuaded to use a structured tool over Grep when both are available. That's an open prompt-engineering / tool-affordance question, not a falsified hypothesis about role workflows.
- **But "ship a snippet, hope for adoption" is dead.** Three trials (X018, X019, X020) all converged on the same pattern: the model uses what it knows (Grep, Read), structured alternatives don't get reached for organically. The lever — if there is one — is more aggressive: hide Grep, mandate code-lookup for symbol queries, or wrap Grep with a wrapper that re-routes symbol-shaped queries to code-lookup.

Tactical follow-ups deferred:

- ~~Re-run with-tool to estimate variance~~ — variance is real but not the interesting question now that the mechanism evidence is sharper.
- **Possible: stronger prompt variant.** Replace the snippet with imperative-with-prohibition phrasing ("Do not use Grep when searching for an exported symbol; use code-lookup mode=symbol or mode=usages instead"). Cheap test, would tell us whether positioning or phrasing was the bottleneck.
- **Possible: route-through-Grep wrapper.** Intercept Grep calls whose pattern matches an exported symbol name and re-route to code-lookup. Doesn't depend on model cooperation. Would establish an upper bound on the mechanism's value — does using code-lookup *actually* drop cost, or is the snippet's behavioral side-effect the only thing helping?

Strategic follow-up:

- **(✓) Investigate what produced the 52% Edit-call reduction in trial 2.** That's the actual cost lever in this data, and it appeared even without code-lookup adoption. Possible candidates: snippet-induced session-shape change, model variance, or bigger/fewer Edits per call. Worth examining the with-tool transcript for editing-pattern differences.
- **(✓) Re-survey the Apr-29 cost landscape (`c-mok4nke6`)** with X018 + X019 + X020 reframed as "preprocessing-and-exposure delivers structured data but the model doesn't reach for it organically." The actionable levers in this family are interception/mandation, not exposure-with-prompting. The interesting uneliminated levers are still in Categories 3 (implementer behavior nudges — X022's territory), 5 (engine pipeline decomposition), and 6 (subagents).
