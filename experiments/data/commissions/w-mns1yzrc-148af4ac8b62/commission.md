## Context

See docs/architecture/detached-sessions.md for the full architecture.

This commission rewires the claude-code session provider to spawn babysitters instead of claude directly, completing the detached sessions feature.

Depends on:
- "Tool HTTP API on the Instrumentarium" — provides the guild-side HTTP endpoint
- "Session Lifecycle Tools on the Animator" — provides session-running/session-record tools
- "Session Babysitter for Detached Anima Sessions" — the babysitter script this provider spawns

## What to Build

### Update the claude-code provider's launch() method

Currently (packages/plugins/claude-code/src/index.ts), the provider's launch() method:
1. Calls prepareSession() — creates tmpDir, writes system prompt, starts in-process MCP server, writes mcp-config
2. Spawns claude directly as a child process
3. Returns { chunks, result, processInfo } — chunks from stdout parsing, result from process exit, processInfo with PID

The new launch() should:
1. Resolve the babysitter script path (compiled output, resolved via import.meta.dirname or __dirname)
2. Build the BabysitterConfig object:
   - sessionId: from the AnimateRequest (pre-generated by Animator's summon())
   - guildToolUrl: from guild config or Instrumentarium API (the Tool HTTP API's URL)
   - dbPath: path to the guild's nexus.db
   - claudeArgs: build the same CLI args as today (model, system-prompt-file, --resume, --print -, --output-format stream-json, --verbose, --setting-sources user, --dangerously-skip-permissions)
   - cwd, env, prompt: from SessionProviderConfig
   - tools: serialize the resolved tools from config.tools — convert each ToolDefinition's Zod params to JSON Schema
   - startedAt, provider, metadata: session metadata
3. Spawn the babysitter as a detached process with stdio: ['pipe', 'ignore', 'inherit']
4. Write BabysitterConfig as JSON to the babysitter's stdin, then close stdin
5. Call proc.unref() so the guild doesn't wait for the babysitter
6. Return { chunks, result, processInfo } that satisfies the AnimatorSessionProvider interface

### AnimatorSessionProvider interface compatibility

The Animator expects launch() to return:
- chunks: AsyncIterable<SessionChunk> — for real-time streaming
- result: Promise<SessionProviderResult> — resolves when session ends
- processInfo?: Promise<Record<string, unknown>> — resolves with { pid }

For detached sessions, these behave differently:

chunks: The babysitter handles streaming by writing transcript chunks directly to SQLite. The guild-side chunks iterable should complete immediately with no items. Real-time output is available via the transcripts book, not the in-memory broadcaster.

result: The provider doesn't directly observe the babysitter's exit (it called unref()). Instead, it should poll the sessions book in Stacks for the session to reach a terminal status. When the babysitter calls session-record, the SessionDoc is updated. The provider polls and resolves the result promise. Use a configurable poll interval (e.g., 5 seconds) with a generous timeout.

Alternatively: the provider can resolve the result promise immediately with a "detached" status, and rely on the Spider's existing tryCollect() to pick up the session result from Stacks. This is simpler but changes the AnimateHandle.result semantics. Discuss which approach is cleaner.

processInfo: The provider doesn't directly know claude's PID (the babysitter spawns claude, not the provider). The babysitter reports the PID via the session-running tool call. The provider can resolve processInfo by polling the SessionDoc for cancelMetadata.pid, or it can resolve immediately with the babysitter's PID as a fallback.

### Tool serialization

The provider needs to serialize Zod tool definitions to JSON Schema for the babysitter config. The MCP SDK has Zod-to-JSON-Schema conversion (it does this internally for MCP tool registration). Use the same approach or zod-to-json-schema directly.

### Existing code preservation

The existing prepareSession() function and in-process MCP server code should be preserved (don't delete them). They may be useful for a future "attached mode" escape hatch for debugging, or for non-detached session types.

### Cancel support

The existing cancel() method (process.kill(pid, 'SIGTERM')) should continue to work. Once the SessionDoc has cancelMetadata.pid (written by the babysitter via session-running), cancel sends SIGTERM to that PID. This works cross-process regardless of parent-child relationships.

## Test Expectations

- BabysitterConfig serialization (unit): tools correctly serialized to JSON Schema, all config fields populated
- Detached spawn (integration): babysitter process starts, receives config via stdin, guild process can exit independently
- AnimateHandle.result resolution (integration): result resolves when session-record is called
- Cancel (integration): cancel sends SIGTERM to claude PID obtained from SessionDoc
- Backward compatibility: existing non-detached code paths are preserved
