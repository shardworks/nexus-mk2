Cost aggregation pulls cross-plugin data through Stacks
D10 recommends direct book reads from spider into animator/sessions. This is already the pattern (see spider/oculus-routes.ts:64 for transcripts), but it does couple Spider to Animator's on-disk schema in more places. If the Animator ever renames or restructures the sessions book, Spider breaks in more places than it does today. An AnimatorApi method (sumCostForSessions(ids: string[]): { costUsd, inputTokens, outputTokens }) would be a cleaner boundary if this coupling becomes problematic. Not worth the abstraction for this commission alone, but a candidate for a future "decouple spider from animator stacks" cleanup if more places grow this dependency.

The brief's token format uses no .toLocaleString() precedent
The brief's example 12,222 input, 12,223 output implies a , thousands separator (en-US locale). Users viewing the dashboard in a non-US locale will see their system's grouping separator if we use Number.toLocaleString() without a locale argument (e.g. 12.222 in German locales). To match the brief exactly for all viewers, the implementation should pass 'en-US' explicitly (n.toLocaleString('en-US')). This is a small formatter-implementation detail the spec writer should name explicitly.
