Reading the framework command pattern (`packages/framework/cli/src/commands/`), the `tool()` factory from `@shardworks/tools-apparatus` is a thin Zod-schema-+-handler wrapper. The coupling audit script doesn't use it (D1 places it outside the CLI). But repurposing `tool()` as a general-purpose 'parameterized handler with validation' building block — useful for any future script that might want a typed param surface — is interesting. Today `tool()` is implicitly coupled to the Instrumentarium / MCP / CLI registration paths. A separable 'lightweight tool() core' that doesn't pull in tools-apparatus's full surface could let analysis scripts share validation logic. Not a refactor for this commission — noted because the audit script's design forced thinking about whether scripts should adopt CLI-grade param validation, and the answer was 'no, it's overkill,' which is itself a signal that the validation tier might be missing.
