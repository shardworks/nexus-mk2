Stacks today exposes only `ensureBook` (idempotent create); there is no API for retiring a book. The cartograph cleanup commission removes three books from the apparatus declaration, but the underlying SQLite tables persist on existing on-disk databases as dead-but-not-dropped storage (`packages/plugins/stacks/src/sqlite-backend.ts:304–` — `ensureBook` only). The brief explicitly tolerates this drift ("There is no migration of historical companion-doc rows"), but it sets a precedent: every future book-retirement commission will leave a layer of dead tables behind, growing the abandoned-data footprint of any long-running guild.

Proposed cleanup commission: extend Stacks's `StacksApi` with a `dropBook(ownerId, bookName)` primitive (and a corresponding backend method) that drops the SQLite table and any associated indexes, with a CDC `delete-book` event for any subscribers that care. Future commissions retiring books would call this from their startup path (or a one-shot migration tool). Concrete payoff: cleaner cartograph cleanup on existing databases; reusable primitive for the surveyor substrate's eventual `books.surveys` retirement (per the brief's §3.7 update to the substrate-owned-things list); enables a `nsg stacks gc` operator tool over time. Affects: cartograph (this commission), surveyor-apparatus (Commission C), any future apparatus that ever retires a book.

Why not concurrent: this is substrate work in `@shardworks/stacks-apparatus`, not in cartograph; the brief explicitly puts substrate work out of scope ("Reworking the `ext` API itself" / similar).
