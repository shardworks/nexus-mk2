`packages/plugins/clerk/src/clerk.ts` exposes `post`, `show`, `transition`, `edit`, `link`, `unlink` — but no `delete(writId)` method. Cartograph's createX methods need writ + companion-doc creation to be atomic; the absence of a delete primitive forecloses the 'create writ via clerk.post then put companion doc and clean up writ on failure' path, since there's no clean-up call to make. This is what forces the more invasive 'replicate clerk.post validation logic inside a stacks.transaction' approach in this commission's decision D15.

**Tactical fix:** Add `ClerkApi.delete(writId): Promise<void>` (or a more conservative `ClerkApi.deleteIfDraft(writId): Promise<void>` that only succeeds when the writ is in `new` phase). This would let plugin authors compose two-step create flows safely without replicating clerk's validation logic. Useful for any future plugin that pairs a writ with a typed companion doc the way astrolabe and cartograph do.

Files affected: `packages/plugins/clerk/src/clerk.ts` (add api method + transaction-aware impl), `packages/plugins/clerk/src/types.ts` (extend ClerkApi interface), test coverage in `packages/plugins/clerk/src/clerk.test.ts`. Consumers: cartograph (this commission), future companion-doc plugins. May want to gate on draft-phase only to avoid accidental deletion of in-flight or terminal writs.
