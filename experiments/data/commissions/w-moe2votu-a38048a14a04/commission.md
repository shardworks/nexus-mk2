The brief documents writ-lifecycle events as firing on `{type}.ready` — `mandate.ready` for a mandate that has been published. The dispatcher binds the relay to the writ that's already in the active state. But the brief also lists a behavioral case: `A standing order { on: "mandate.ready", run: "summon-relay", with: { role: "artificer", prompt: "..." } } fires on a matching event; the dispatcher invokes summon-relay with params: { role: "artificer", prompt: "..." }; an anima session is launched bound to the event's writ.`

That assumes the writ is already in `open` (active) when the event fires. If a future code path emits `mandate.ready` from inside `transition(... 'new', 'open')` (the publish step), the writ is in `open` by event-fire time and no relay-side promotion is needed. But if a plugin emits `mandate.ready` for a `new` writ as a courtesy (“I need someone to look at this draft”), the relay would need to either decline or transition the writ first.

Surfacing the question now: the relay should explicitly assume `mandate.ready` means `the writ is in the active state already` and treat anything else as a programming error (throw, with a message naming the writ's actual phase). Encoding that assumption would trap upstream emission bugs early.
