The recommended approach in D16/D17 introduces a block type `animator-paused` whose checker polls Animator status. Future provider-boundary states — operator pause (sibling click), per-model rate limits, cross-provider fallback, maintenance windows — could all benefit from the same pattern with different condition payloads. Suggestion for future design: make the block type's condition carry the specific pause reason so operators triaging blocked engines can distinguish at a glance. Out-of-scope specialization here, but worth reserving the shape (e.g. `condition: { reason: 'rate-limit' | 'operator' | … }`) rather than a fully empty condition.
