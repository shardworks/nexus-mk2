`docs/architecture/plugins.md` states twice that "`nsg init` installs a default plugin set" (lines 9 and 547). The actual `packages/framework/cli/src/commands/init.ts` writes `plugins: []` via `createInitialGuildConfig`, and the `init.test.ts` regression even asserts `assert.deepEqual(config.plugins, [])`. The init handler then prints `nsg plugin install @shardworks/nexus-stdlib` as a follow-up step, but that aggregator package is not in this repository.

This stale-docs issue intersects the C3 commission's S5 ("Default install") and is the load-bearing reason that S5 cannot be satisfied as the brief literally describes — there is no default-plugins list to add to. Resolution options: (a) update the docs to reflect the manual-install model, (b) introduce a `defaultPlugins` mechanism in `init.ts` that pre-populates `plugins` and runs `npm install` for them, or (c) audit whether `@shardworks/nexus-stdlib` exists as an out-of-tree aggregator and rework the docs to point at that. Recommend (a) as the smallest fix; (b) is a larger framework-level decision.
