**The discrepancy.** `docs/architecture/petitioner-registration.md` §9 sketches a standing-order recipe with `with: { filterExtSource: 'vision-keeper.snapshot', filterPhase: 'cancelled' }` and explicitly flags in §15(g) that these filter keys 'depend on Clockworks's filter capabilities, which are out of scope here.' The standing-order-validator (`packages/plugins/clockworks/src/standing-order-validator.ts`) allows only `on/schedule/run/with` top-level keys and treats `with:` as a free-form record passed to the relay. Filter keys would be silently absorbed as relay params today — exactly the validator's purpose is to reject typos.

**Why this matters.** Every petitioner that follows the Channel-1 recipe writes the same 5-line in-handler filter (`change.type === 'update'` && `change.entry.phase === 'cancelled'` && `change.entry.ext?.reckoner?.source === 'X'`). Either the contract document promises filter keys it cannot deliver, or Clockworks should grow a filter-DSL that pushes these into the dispatcher.

**Proposal.** Either revise the contract document's example to remove the `filterExtSource`/`filterPhase` sketch and show the in-handler-filter pattern (vision-keeper's actual implementation will be the canonical example), OR extend Clockworks's standing-order shape with a typed filter block (`filter: { extSource?, phase?, source? }`) that the dispatcher applies pre-relay-invocation. The latter pulls a filter-DSL into the framework; the former realigns docs with code.

**Atomicity.** Documentation realignment is a small standalone commission. The DSL extension is a larger separate commission — kept atomic only if the simpler docs-realignment is ruled inadequate.
