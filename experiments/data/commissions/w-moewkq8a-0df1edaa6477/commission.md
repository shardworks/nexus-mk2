Per `docs/reference/event-catalog.md` and `packages/plugins/clockworks/src/signal-validator.ts`, the reserved-namespace list (`anima.`, `commission.`, `tool.`, `migration.`, `guild.`, `standing-order.`, `session.`, `schedule.`) does not include `reckoner.` or `reckoning.`. This means an anima could theoretically signal a forged `reckoning.declined` event via the `signal` tool if such an event were ever declared in `clockworks.events`. Same gap exists for `book.*` (event-catalog.md notes this explicitly).

Decision D8 recommends NOT emitting named `reckoning.<outcome>` events today — subscribers wire to the auto-wired book event — so this gap is not load-bearing in v0. But if a future commission earns the named-events surface (D14 names this as an open question), the validator should be updated at the same time to add `reckoning.` to `RESERVED_EVENT_NAMESPACES` so animas can't forge them. Specifically: edit `packages/plugins/clockworks/src/signal-validator.ts` `RESERVED_EVENT_NAMESPACES` array (the constant is exported and the test asserts it).

This is a coupled change — don't add `reckoning.` to the reserved list before adding the named-events emitter, and don't add the emitter without simultaneously reserving the namespace. Note this in the Reckonings book doc's Open Questions section so the future implementer remembers.
