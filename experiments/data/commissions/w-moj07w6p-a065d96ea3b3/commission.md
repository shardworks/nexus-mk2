D4 selected an empty interface for `CapacitySnapshot` in v0. The brief calls this a stub: 'Future commissions fill in real slot/budget tracking when a capacity-aware scheduler ships.' Real fields would likely include in-flight rig count, token-budget remaining, per-source quotas, and per-domain caps — most of which require new accessors on the Spider api (in-flight rig count) or new Stacks queries (recent token-spend aggregation).

When a capacity-aware scheduler is on the table, design questions surface: where does CapacitySnapshot get sampled (inside the helper from D20, or via a separate api method on the Reckoner)? Does sampling happen synchronously per evaluation or is it cached per tick? Are the values point-in-time or trailing-window? Track for the capacity-aware scheduler design commission — not earned by the substrate.
