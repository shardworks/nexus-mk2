`packages/plugins/clockworks/src/summon-relay.ts` reads `writ.status?.clockworks?.sessionAttempts`, increments, and writes back without compare-and-swap. Under the at-most-twice cross-process dispatch contract this commission introduces, two concurrent `processEvents` callers each dispatching the same `mandate.ready` event would both see the same `sessionAttempts` value, both increment to `n+1`, and both launch a session — exceeding `maxSessions` by one. The persisted counter ends at `n+1` (last writer wins) so the breaker subsequently behaves correctly, but the concurrent-overlap window can leak one extra session past the cap.

The substrate-side fix path is a Clerk-side compare-and-swap (Clerk supports `transition` with a `from` precondition; an analogous `patchStatusSlot({ if: { sessionAttempts: n } })` would work) or wrapping the read+increment+write in a Clerk transaction. Out of scope for the doc/contract follow-up but worth filing now under the new contract — the contract makes this a known weakness instead of an undocumented assumption.

Follow-up scope: tighten the increment to be atomic, then re-audit the test in `summon-relay.test.ts` for a concurrent-dispatch case. No relay-author API change needed (the behaviour is internal to the stdlib relay).
