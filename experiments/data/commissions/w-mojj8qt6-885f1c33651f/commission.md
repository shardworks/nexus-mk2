The framework CLI auto-builder (`packages/framework/cli/src/program.ts:detectPositionalId`, lines 63-73) only promotes a single required string param to a Commander positional argument when it is named `id` or ends with `Id`. This forces the brief's `nsg vision apply <slug>` invocation shape to either (a) rename the natural `slug` param to something id-like (semantic mismatch — a slug is not an id), (b) add explicit positional-arg metadata to the `tool()` factory (a wider framework change), or (c) silently diverge to `--slug <slug>` (breaks the brief's literal CLI shape).

Decision D4 in this commission picks the framework-side detector generalization as the cleanest path. The change itself — relax `detectPositionalId` to promote any single-required-string param that isn't ambiguous with another id-like one — is small, but it lives in `packages/framework/cli/` and touches a heuristic that is shared by every plugin-contributed tool. A quick audit is warranted: any tool today with a single required string param NOT named `id` would gain a positional argument it didn't previously have. The existing tools that currently use `id` or `*Id` keep the same behavior; the question is what other tools exist with non-id-like single required strings.
