Decision D7 chose per-source indexing on `ScheduleEntry`, which D20 propagates into the `clockworks.timer` event payload (`packages/plugins/clockworks/src/scheduler.ts:248`). Today the only consumers of that payload are integration tests (`scheduler-integration.test.ts` lines 252–259), and they assert `orderIndex === 0` — which the per-source change preserves for the operator-only case.

But if a future consumer (the Reckoner, a future log shipper, or an external observer) starts correlating `orderIndex` between the synthesized `clockworks.timer` row and `guild.json`'s `clockworks.standingOrders` array, the kit-source case must explicitly document that orderIndex points into the kit's own array — not into guild.json. The new `source` field on the payload (D20) is the disambiguator, but consumers that ignore it will silently mis-correlate. Worth surfacing in the architecture doc's payload spec (the brief's docs already mention orderIndex in the `clockworks.timer` payload section but should call out the per-source semantics explicitly).
