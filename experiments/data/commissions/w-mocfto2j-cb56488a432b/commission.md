`packages/plugins/spider/src/spider.ts:1493-1501` currently treats every `session.status === 'failed' || 'timeout'` as `failEngine(… retryable: true, detail: 'Session failed: …')`. The rate-limit commission has to introduce a branch here, but the existing pattern shows a latent issue: the classification `retryable: true` is blanket-applied to all session crashes with no discrimination. A future observation: provider timeouts likely should be retryable in a way that quota exhaustion should not be (even before the rate-limit work). This commission shouldn't fix that, but the tryCollect switch is the correct place to later add more surface-level branching as more signal types are surfaced.
