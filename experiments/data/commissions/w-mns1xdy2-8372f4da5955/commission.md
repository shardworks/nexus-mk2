## Context

See docs/architecture/detached-sessions.md for the full architecture.

The Detached Sessions architecture decouples anima sessions from the guild process. Session babysitters (detached processes that host claude sessions) need to report session lifecycle events back to the guild via HTTP tool calls. This commission adds the tools that accept those reports.

Depends on: "Tool HTTP API on the Instrumentarium" (the tool server that makes these tools callable over HTTP).

## What to Build

### 1. session-running Tool

A new tool registered by the Animator that records the initial "running" SessionDoc. Called by the session babysitter immediately after spawning claude.

Parameters:
- sessionId (string, required) — the session ID (pre-generated by the Animator at dispatch time)
- startedAt (string, required) — ISO timestamp of session start
- provider (string, required) — provider name (e.g., "claude-code")
- conversationId (string, optional) — for --resume sessions
- metadata (object, optional) — session metadata (writId, engineId, etc.)
- cancelMetadata (object, optional) — includes { pid } for cross-process cancellation

Handler: writes a SessionDoc with status: 'running' to the sessions book via sessions.put(). This fires CDC (Laboratory observes session start).

This replaces the current in-process recordRunning() function for detached sessions. The existing in-process path should still work for any non-detached usage.

callableBy: ['infrastructure'] — this is not a patron or anima tool.

### 2. session-record Tool

A new tool registered by the Animator that records a terminal session result. Called by the session babysitter when claude exits.

Parameters:
- sessionId (string, required)
- status (enum: 'completed' | 'failed' | 'timeout', required)
- exitCode (number, required)
- error (string, optional)
- costUsd (number, optional)
- tokenUsage (object, optional) — { inputTokens, outputTokens, cacheReadTokens?, cacheWriteTokens? }
- output (string, optional) — final assistant text
- providerSessionId (string, optional) — claude's session ID for --resume
- conversationId (string, optional)

Handler logic:
1. Check if session was cancelled (read current SessionDoc — if status is 'cancelled', don't overwrite; just record transcript if present). This mirrors the existing cancellation check in animate().
2. Build SessionDoc from params, write via sessions.put().
3. If transcript is provided, write TranscriptDoc via transcripts.put().
4. Fires CDC for both writes.

Note: The transcript may NOT be passed via this tool call — the babysitter writes transcripts directly to SQLite for real-time streaming (see Session Babysitter commission). This tool handles the session status + metadata. But include transcript as an optional param for flexibility.

callableBy: ['infrastructure']

### 3. DLQ Drain on Startup

On Animator apparatus startup, scan .nexus/dlq/ directory (relative to guild home) for JSON files. Each file contains a session-record payload that a babysitter couldn't deliver (guild was down). Process each file through the session-record handler logic (writes to Stacks, fires CDC). Delete the file after successful processing. Log warnings for any files that fail to process.

Create the .nexus/dlq/ directory if it doesn't exist.

### 4. Orphan Recovery on Startup

On Animator apparatus startup (after DLQ drain), query the sessions book for all sessions with status = 'running'. For each:
- Read cancelMetadata.pid from the SessionDoc
- Check if the process is alive: process.kill(pid, 0) — if it throws ESRCH, the process is dead
- If dead: patch the SessionDoc to status: 'failed' with error: 'Session process died unexpectedly (orphaned)', endedAt: now
- If alive or no PID: skip (session is still legitimately running, or pre-detached session without PID)

This handles the case where both the guild AND the babysitter died (machine crash, OOM). Without this, sessions would be stuck in 'running' forever and the Spider would wait indefinitely.

## Technical Notes

- The callableBy: ['infrastructure'] designation is new. The Instrumentarium's tool resolution currently supports 'patron' and 'anima'. Adding 'infrastructure' as a caller type requires checking that the resolve/filter logic handles it correctly. If this is too invasive, use callableBy: ['anima'] as a pragmatic alternative — the authorization boundary is enforced at the Tool HTTP API session layer, not the callableBy filter.
- The DLQ file format should be a JSON object matching the session-record tool's params schema. Filename: {sessionId}.json.
- Orphan recovery should run after DLQ drain so that DLQ'd results are processed before orphan scanning (a DLQ'd result might resolve what would otherwise look like an orphan).

## Test Expectations

- session-running tool: writes correct SessionDoc, handles duplicate calls gracefully
- session-record tool: writes SessionDoc + TranscriptDoc, respects cancelled status, handles missing session gracefully
- DLQ drain: processes files, deletes after success, logs on failure
- Orphan recovery: marks dead-PID sessions as failed, skips alive sessions, skips sessions without PID
