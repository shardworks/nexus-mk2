The hold-reason 'rate-limit' is set only when tryCollect sees a terminal `session.status='rate-limited'`. A fresh-pause detected proactively (brief mentions the Animator owns rate-limit state) wouldn't place the gate on engines that weren't running at the pause boundary. For the initial model this is fine — `isAnimatorPaused` gate on trySpawn handles the fresh-pause case — but under load with many rigs there will be brief windows where some pending engines don't yet carry holdReason but nonetheless cannot dispatch. Dispatch predicate's external-gate check against Animator state covers this correctly if implemented as a live-read per-tick.
