`docs/reference/event-catalog.md` and `docs/architecture/clockworks.md` disagree about the framework event list. Event catalog describes `commission.posted`, `commission.session.ended`, `commission.completed`, `session.started`, `session.ended`, `session.record-failed`, and `{type}.stuck`. Clockworks.md's framework-events table lists `commission.posted`, `commission.state.changed`, `commission.sealed`, `commission.failed`, `anima.instantiated`, `anima.state.changed`, `anima.manifested`, `anima.session.ended`, `{type}.ready`, `{type}.completed`, `{type}.failed`, `tool.installed`, `tool.removed`, `migration.applied`, `guild.initialized`, `standing-order.failed`. Overlapping but not identical sets.

None of these events are emitted by code today. Before task 3 (`signal` tool) and task 4 (runner) ship, the canonical framework-event list should be reconciled into a single source of truth. Worth a small commission before the Clockworks behavioural tasks land — otherwise task 4 has to guess which catalog to implement.
