`packages/plugins/claude-code/src/babysitter.ts` `runBabysitter` (around line 887–898) and `writeTranscript` (around line 528–535): every NDJSON message that grows the transcript triggers a full `INSERT OR REPLACE` of the entire transcript blob keyed by sessionId in `books_animator_transcripts`. For an N-message session the total bytes written are O(N²) of the transcript size — the message-1 write costs 1 unit, message-2 costs 2, …, message-N costs N. For long sessions (hundreds of tool calls) this is real I/O cost.

The current shape is *correct* and *spec-compliant* (the transcripts book is single-row-per-session per the architecture spec, and WAL mode lets concurrent readers see partial state). It is also load-bearing for the way other consumers (Oculus, dashboards) read transcripts. So this is **not a correctness fix**.

What to investigate: whether an append-only per-message row schema (rows keyed by `(sessionId, messageIdx)`) would let the babysitter only write the *new* rows, with a query-time aggregator producing the same shape readers see today. The trade-offs are real (read query gets more complex; the books abstraction may not fit per-message rows naturally), so this is a **decision-bearing** investigation, not a one-shot fix. Out of scope for the current complexity diagnosis; surface it as a separate “write-amplification” commission.
