The retry cap lives as a compile-time constant in `@shardworks/clockworks-retry-apparatus`. Spider reads `SpiderWritStatus.retryable` (which is produced by Spider's own `failEngine`). The Reckoner (per D16) reads `ClockworksRetryApi.maxAttempts` and `rigs.count` to decide emit policy. Three apparatus now key on the same cap; a future config-driven cap (e.g. `clockworks-retry.maxAttempts`) would ripple through all three. Not a bug — the current single source of truth is honored — but worth documenting the read-coupling so the cap can move cleanly.
