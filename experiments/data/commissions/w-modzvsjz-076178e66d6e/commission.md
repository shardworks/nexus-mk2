Per D3/D4, `tick <id>` uses `events.get(id)` (PK lookup, free) for the pre-check and then calls `processEvents({ eventId: id })` which would translate to `where: id = X` inside `runDispatchSweep`. The events book already indexes `id` (it's the PK), so no schema change. But `processEvents({ max: 1 })` (the no-arg `tick` form) translates to `where: processed = false orderBy id asc limit 1` — the existing `(processed, firedAt)` composite index covers `processed` filter + sort by `firedAt`, but not by `id`. Worth verifying the SQLite query plan picks up the `processed` single-column index for this path; if not, add an `(processed, id)` composite index. Tiny optimization; skip if benchmarking does not show a need.
