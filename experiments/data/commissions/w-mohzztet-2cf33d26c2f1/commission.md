Decision D7 in this plan recommends mirroring the watcher's `clockworks/events` carve-out in the events-kit declaration so the declared set equals the emitted set (no doc lie). The trade-off: an anima could in principle `signal('book.clockworks.events.created', ...)` because that name is not declared, hitting the merged-set check's "not declared" rejection rather than the framework-owned rejection. Today's behaviour is identical (no declaration there either) and the spoofed event would land as a custom-event row only if the operator declared it under `guild.json clockworks.events` — which is the existing as-is gap.

After this commission lands, the gap surface shrinks: every `(owner, book, verb)` tuple of every plugin-contributed book becomes plugin-declared, leaving only the carve-out tuples (book.clockworks.events.{created,updated,deleted}) plus any tuple keyed off a book contributed after start (which the substrate prevents). The remaining surface is small enough that an explicit static-suffix declaration `book.clockworks.events.{created,updated,deleted}` in the events kit would close it entirely. Worth a follow-up commission to evaluate whether the closure earns its keep.
