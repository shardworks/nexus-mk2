`packages/plugins/stacks/docs/specification.md:379-389` and `docs/architecture/apparatus/stacks.md:245-249` both describe Phase-1 handler transaction binding as managed by `AsyncLocalStorage`. `grep -r AsyncLocalStorage packages/plugins/stacks` is empty — the substrate (`packages/plugins/stacks/src/stacks-core.ts:105`) carries transaction state on a single `activeTx: ActiveTransaction | null` instance field. Functionally equivalent today because the substrate is single-flight per `StacksCore` instance, but the spec is making a claim the code does not back.

This commission (D13 = minimal-introduction) introduces `AsyncLocalStorage` only for the new Phase-2 re-entry counter. The wider retrofit — moving the existing transaction-binding path onto `AsyncLocalStorage` so the spec's claim is honoured — is intentionally out of scope. A follow-up should either (a) migrate `runTransaction` and the doPut/doPatch/doDelete paths to use the same `AsyncLocalStorage` so the spec matches the code, or (b) update both specs to describe the actual `activeTx` instance-field implementation. Option (a) is the more durable choice if Stacks ever needs to support multi-flight execution (e.g. concurrent transactions in a future async backend); option (b) is the cheap fix if single-flight is the long-term contract.
