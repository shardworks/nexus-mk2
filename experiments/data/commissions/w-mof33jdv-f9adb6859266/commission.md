`FabricatorApi.listEngineDesigns()` returns `EngineDesignInfo[] = { id, pluginId, hasCollect }[]` (`packages/plugins/fabricator/src/fabricator.ts` lines 226–233, 309–320). The `engine-designs` tool (`packages/plugins/spider/src/tools/engine-designs.ts`) is a thin pass-through. Neither surface exposes the design's `retry` field or, after this commission lands, the resolved retry config (override-merged-on-design).

With guild-level overrides shipping, operators have no programmatic way to inspect the *effective* retry policy without reading guild.json by hand and mentally merging it with the engine source. This is a separate observability gap from obs-2 (rig-show CLI rendering); this one is about the registry-introspection surface itself.

Follow-up: extend `EngineDesignInfo` to include an optional `effectiveRetry: { maxAttempts: number, backoff: { initialMs, maxMs, factor } }` field, populated by Fabricator on listEngineDesigns. Either Spider feeds the override map into `FabricatorApi` (via a new method like `setOverrideResolver`), or Fabricator continues to return the bare design retry and Spider exposes an enriched view via its own tool that consults both layers. The umbrella commission's obs-9 covers the same observability gap pre-overrides; this observation extends the gap to the override-aware case.
