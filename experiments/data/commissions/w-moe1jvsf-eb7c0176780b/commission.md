When a relay is broken (e.g. throws on every invocation) and a high-frequency event keeps firing, the events book can accumulate many SOF rows in tight succession — each one is a real failure, but the resulting alert volume can swamp an operator. The brief explicitly punts on this ('Automatic disabling of a chronically-failing standing order: Guild can write an order that watches failure rates and disables siblings if it wants this'), but it would be worth exploring (a) a stdlib relay that watches failure rates, or (b) a Clockworks counter table that records SOF frequency by `(on, run)` tuple so operator-facing surfaces can rate-limit alerts.

Tactical detail: this is a follow-on capability, not a bug. A future commission might add a `clockworks/standing_order_health` book keyed by `(on, run)` with a rolling failure count and a last-failure timestamp. The Reckoner-tick or a dedicated stdlib relay could read it. Filing as an observation so the idea isn't lost when SOF starts producing real failure traffic.
