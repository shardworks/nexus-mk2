Root cause for the Apr-26 incident was `git checkout HEAD~1` issued mid-flow during review/revise. Once seal absorbs detached-HEAD via reattach, the failure mode becomes invisible end-to-end — which is also a loss: the `[scriptorium]` log line is the only trace that something unusual happened. A complementary engine-side guard (run `git symbolic-ref HEAD` at engine entry; warn or fail if detached) would catch the upstream cause closer to where it was introduced. The brief explicitly excludes this from scope ("No detached-HEAD prevention… does not change implement/review/revise engine behavior") but it's the obvious follow-on commission for closing the loop on the failure mode altogether.

Files: `packages/plugins/spider/src/engines/implement.ts`, `review.ts`, `revise.ts`. The implement-loop pre-step would be a natural single point of insertion if a shared guard is preferred over per-engine duplication.
