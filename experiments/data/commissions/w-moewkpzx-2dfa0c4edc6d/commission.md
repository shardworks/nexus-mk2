Click c-modaqnpt ('Deferred-petition metadata shape') concludes that 'Escalation policies (auto-decline after N defers, nudge priority on repeat defer) are Reckoner config, not petition-shape.' The `defer_count` field on the petition record drives this. The Reckonings book duplicates `defer_count` semantics implicitly — a per-petition timeline of `outcome='deferred'` rows IS the count. If the auto-decline-after-N config is read against the Reckonings book (count rows) instead of the petition record's `defer_count` field, the two sources can diverge under retention pruning (D6 future archival): old deferred Reckonings rows get archived, the count drops, the auto-decline never fires.

The Reckoner core commission needs to settle which is authoritative for the auto-decline gate: the petition record's `defer_count` field (current-state materialized view, won't be affected by Reckonings retention) or the Reckonings book (event journal, will be affected). Recommended: petition record is authoritative; Reckonings is the audit-log derivative. Note this in the Reckonings book design doc as a hand-off to the Reckoner core spec.

This is a one-line cross-reference in the doc, not a major design concern — but it deserves explicit treatment so the Reckoner core implementer doesn't accidentally read defer counts from the Reckonings book.
