## By Experiment

### X006 — Guild Metaphor

The dispatch → commission rename follows the exact pattern established by implement → tool: a technical jargon word with heavy prior meaning causing repeated double-takes, resolved by the metaphor's own vocabulary. Notably, he didn't say "commission felt more evocative" — he said dispatch didn't *fit*, and commission was "the more direct verb from the metaphor." The metaphor is doing practical disambiguation work, not just aesthetic work.

He framed this as the friction-accumulates-before-fixing pattern, not early detection. The dual-register test is still running in the background but vocabulary problems need to build up before they get fixed. Token cost is now an explicit factor in when he pulls the trigger on large renames — adding a new friction layer on top of the vocabulary friction.

### X007 — First Contact

The first autonomous commission is imminent. Pre-dispatch observations:

- **Sean's expectations:** He expects failure. Specifically: implementation-layer glitches (null pointers, data shape mismatches) from Coco-written code that wasn't reviewed. He trusts the architecture, not the details.
- **Commission selection strategy:** Toy apps, TUI games, "hello world" scripts first. Nothing touching the real system. "No horse in the race" — clean observation conditions.
- **Intervention policy:** No intervention planned. He'll let commissions run to completion regardless of outcome.
- **His framing:** "Smoke testing" — not expecting success, just checking where things break.

This is methodical and un-romantic. No excitement about the guild "finally coming alive" — he's treating it as an engineering smoke test. The "no horse in the race" strategy is smart for clean observation but also means the first commissions won't tell us how he evaluates work he cares about.

### X008 — Patron's Hands

Token quota is now a tangible resource constraint. 23% of weekly quota consumed in one day. Sean framed it as "lingering unease" — not driving decisions yet, but present. This adds a new dimension to the Coco-as-implementer pattern: it's not just about the tight loop, it's about the resource cost of interactive implementation. Large cross-cutting refactors through Coco are expensive. If autonomous agents can do similar work at lower token cost (or in parallel), that's a concrete functional advantage — which maps to one of the thresholds he named in the previous interview.

The "I've never seen AI coding agents create complex code that didn't have glitches" quote is relevant: his confidence calibration for Coco-written code is informed by general prior experience with AI coding agents, not just this project. He doesn't have higher trust in Coco because of the tight loop — his trust ceiling is set by category, not by interaction mode.

## Cross-Cutting Observations

The multi-agent oversight hypothesis is interesting: he's explicitly naming the hope that agents checking each other's work produces better quality than single-agent implementation. This is a theory about the guild's architecture that he's carrying into the first dispatch. It may shape how he evaluates outputs — if the first commission fails on implementation details, does he attribute it to "the guild isn't running properly yet" (no oversight loop) rather than "AI agents can't do this"?

The resource constraint (token quota) is a new variable that hasn't appeared in previous interviews. It creates subtle pressure toward autonomous dispatch even before the functional thresholds are fully met — if Coco keeps consuming quota at this rate, the calculus may shift.
