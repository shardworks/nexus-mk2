# Session Summary

## What we did

- Investigated why the astrolabe planning pipeline has been silently degraded, starting from Sean's hypothesis that tool naming (nice names vs `mcp__nexus-guild__*`) was confusing the anima.
- Confirmed the MCP naming scheme is `mcp__nexus-guild__<tool>` by reading `packages/plugins/claude-code/src/index.ts` and `mcp-server.ts` in the framework repo and cross-checking against real tool-use events in `experiments/data/transcripts/`.
- Pulled 51 astrolabe anima sessions (22 reader / 17 analyst / 12 spec-writer) from `books_animator_transcripts` in `/workspace/vibers/.nexus/nexus.db` and enumerated every `tool_use` block. Zero calls to any `mcp__nexus-guild__*` tool across the entire history.
- Found explicit confirmation in the anima text: multiple sessions say "the Astrolabe tools aren't available" and pivot to writing directly to the guild SQLite database via `better-sqlite3` from `node_modules`.
- Discarded the naming hypothesis; ran a thorough review of all 60 plugin-contributed tools (name, permission, callableBy) plus all defined roles (guild-level in `/workspace/vibers/guild.json` and kit-level via `astrolabe.sage`).
- Identified the real bug: two incompatible permission naming conventions across plugins (bare `read`/`write` vs plugin-qualified `clerk:read`/`astrolabe:write`/`spider:write`). The Instrumentarium's `matchesPermission` compares `grant.level === permission`, which fails whenever a tool uses the qualified form against a non-wildcard grant.
- Traced why this has been invisible: every live role uses wildcards (`steward: *:*`, `artificer: tools:*`) except the kit-registered `astrolabe.sage`, which uses strict + non-wildcard grants. Sage is the only role in the system that exercises the buggy path, and it resolves to the empty tool set.
- Answered Sean's direct question about `session-record` / `session-running`: looked at `packages/plugins/tools/src/tool.ts` and confirmed the bare-level form (`permission: 'write'`) is the documented convention; `pluginId` is attached at registration time, so no prefix is automatic and none is meant to be.
- Drafted the mandate at `.scratch/mandate-tool-permission-convention.md` (Option 1: normalize qualified-form tools to bare-level). Iterated twice at Sean's request — added a convention-codifying test, and an explicit "Affected plugins" summary enumerating clerk (11 tools), astrolabe (7), spider (8), plus a hands-off list of plugins that must not be touched.
- Attempted to post via `bin/commission.sh`; Arbor startup logging leaked into stdout and broke JSON parsing inside the script. Surfaced the error and stopped per standing orders on CLI breakage. Sean posted the commission manually.

## Decisions made and/or deferred

- **Chose Option 1** (normalize to bare-level) over Option 2 (make the matcher accept either form). Rationale: Option 1 is the documented convention, the majority of plugins already follow it, and codifying the inconsistency in the matcher would mask future drift.
- **Chose to add a registration-time guard** (throw on permissions containing `:`) rather than leave this as a pure fix. Prevents the class of drift from re-emerging.
- **Deferred** investigating the `commission.sh` Arbor-stdout breakage. Sean worked around it by posting manually.
- **Did not** touch `sage.md` prose, role permissions, or the grant-string format — fix is strictly on the tool-definition side.

### Notable moments

- **Sean on probing the first hypothesis without chasing agreement:** the initial framing ("maybe the nice names don't line up with what the anima sees") was plausible and I confirmed the naming scheme quickly. But on inspecting actual transcripts, the `mcp__nexus-guild__*` count was flatly zero — not "some misses, some hits under the full name." That forced a rejection of the hypothesis and a deeper dig. Notable because it's a clean example of letting the evidence override a narrative that already "made sense." If I'd stopped at "yep, it's the naming" and drafted a prose fix to `sage.md`, the real bug would still be live.
- **Sean's "why no plugin prefix" question on session-record/session-running:** pushed me to read `tool.ts` properly and surface the documented convention, which was the keystone for the whole diagnosis. Before that question I was treating "qualified vs bare" as a symmetric inconsistency; after it, the bare form is clearly canonical and everything else is drift. The question reshaped the recommendation from "pick a side" to "there is a side and three plugins are wrong."
- **The SQLite workaround:** multiple spec-writer animas, finding the tools missing, wrote `const Database = require('.../better-sqlite3')` and patched the guild DB directly. Notable as an ethnographic data point about anima resourcefulness — and as a warning sign that silent tool-resolution failures produce silent-but-wrong behavior, not visible crashes.
