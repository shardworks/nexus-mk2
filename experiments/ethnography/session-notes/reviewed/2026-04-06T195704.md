## What we did

- Collaborated on naming for the new engine state: explored "paused," "detained," "blocked," "held," "awaiting." Settled on **"blocked"** after stress-testing candidates in logs, dashboards, and operator conversations.
- Designed the block-resolution architecture: compared two approaches — (A) Spider polls by restarting engines (engines self-check) vs. (B) block registry with typed condition checkers. - Designed ownership model: Spider owns the block-checking loop; block types are registered guild apparatus with lightweight checkers, condition schemas, and poll interval hints; Spider validates block types at block time (fail-fast).

## Decisions made and/or deferred

- **Decided: "blocked"** as the status term. "Detained" was etymologically perfect but failed the operator-ergonomics test (dashboard readability, log naturalness, state transition vocabulary).
- **Decided: Option B architecture** — block types as registered apparatus with condition checkers, not engine self-check. Key driver: LLM cost avoidance for trivial condition checks.
- **Decided: Spider owns block checking** — no separate "block watcher" component. Spider already owns engine lifecycle; block checking is a natural extension.
- **Decided: fail-fast validation** — Spider rejects blocks on unregistered types or schema-mismatched conditions.
- **Deferred to planner:** crawl loop phase ordering, collect() return type, storage shape, crawl-continual idle detection, prior-block context passing.
