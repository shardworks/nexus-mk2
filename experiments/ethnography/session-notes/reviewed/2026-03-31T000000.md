## Decisions made and/or deferred

- **Keep "Kit"** — Sean rejected "Craft" and other alternatives; Kit stays
- **Everything is a plugin** — no built-in apparatus; even Clockworks, Walker, Surveyor are plugins. Guild core is a loader + type contracts only.
- **Single kit per plugin** — no array of named kits; they'd just be merged anyway
- **`start(ctx)` / `stop()`** — kept over `activate`. Explicit `stop()` is better than burying shutdown in an event subscription.
- **Flat `requires: string[]`** — no two-mode syntax distinguishing `{ interface }` from `{ plugin }`. Framework classifies internally; declaration is just a string.
- **`ctx.plugin<T>(name)`** — service locator over DepContainer. Simpler interface; validation against `requires` enforced at startup gives the same failure-mode benefits.
- **Validation at startup** — missing or undeclared dependencies fail before any `start` is called, never mid-commission.
- **Synchronous start/stop** — fire-and-forget async is possible by not awaiting inside the handler; framework doesn't await return values.
- **Static contributions in manifest** — engine designs, tools, `provides` object reference all declared in the manifest; dynamic wiring in `start()` only for things that require a running apparatus.
- **`plugin:initialized` via snapshot + subscription** — no priority fields, no init phases; `ctx.plugins()` for past, `ctx.on("plugin:initialized")` for future.
- **Deferred:** Tier 1 interface aliases (e.g. `ctx.clockworks` shorthand) — concept exists implicitly but not specified.
