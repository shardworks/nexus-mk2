- **Identified critical bugs in current system:** artificers can't record strokes because no jobs exist (the commission pipeline skips the work hierarchy entirely). The clockworks has hardcoded commission logic that should be generic.

## Decisions made and/or deferred

- **Rigid hierarchy → flexible typed work items.** Replace work/piece/job/stroke with a single `work_items` table. Types are guild-defined. Parent/child is optional. One completion rollup mechanism.
- **Strokes eliminated as a concept.** Atomic progress records become child work items (type "step" or whatever the guild calls them). One data model all the way down.
- **Every summon gets a work item.** Framework auto-synthesizes for non-work-item events. Gives uniform resumption, progress tracking, and audit.
- **Prompt templates on standing orders.** Replace hardcoded commission logic in clockworks with template hydration from event payloads and work item lookups.
- **`complete-session` as universal signal.** One tool, one meaning: "I'm done with my part."
- **Deferred:** Migration path, prompt template traversal depth, step event noise, re-summon circuit breakers.

### Notable moments

- **"for llms, is 'more maintainable' mean 'refactored for no duplication' or 'everything is duplicated to reduce blast radius'?"** — Sean challenged the assumption that DRY is good for agent-maintained code. Led to X012 experiment. Notable because it questions whether foundational software engineering principles are human-specific.

- **"i don't think we should be able to 'work on' a commission, now that it's a delivery mechanism"** — Triggered the entire design session. Sean identified that commissions had become conflated with work items, violating the separation the docs described.

- **"does it make sense to always require a job?"** followed by **"think through a wide variety of hypothetical event triggers that would summon an anima"** — Sean pushed back on Coco's position that all summons need jobs. The enumeration of scenarios (foreman, midnight inspector, triage sage) proved only 1 of 11 cases naturally wanted a job. This was the key insight that broke the rigid hierarchy.

- **"what if we didn't require strokes to have a job? what do we lose, what stops making sense?"** → **"what if all work_items need a type?"** → **"what if we synthesize a work item for every summon?"** — A cascade of simplifications, each building on the last, arriving at a design significantly simpler than the starting point. Notable because Sean drove each simplification with a question, not a prescription.

- **"i don't think work items should have a 'commissionId', should they?"** — Late-stage feedback that inverts the relationship: commission points to its root work item, not every work item pointing back to a commission. Cleaner data model.
