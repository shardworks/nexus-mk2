Short, operational session. Sean asked Coco to patch the guild DB to restart a failed rig; the diagnosis opened a small window onto a real bug in the reader/inventory pipeline.

## What we did

- Investigated `rig-mnxy3qqq-4de722e1` (writ `w-mnxy3q64-5a094ccf5be7`, "Update detached-session implementation" Phases 2–3). Rig was `failed`: `inventory-check` failed with *"Plan … has no inventory — reader stage did not produce output"*; everything downstream (`analyst`, `decision-review`, `spec-writer`, `spec-publish`, `seal`) was `cancelled`.
- Located the guild DB at `/workspace/vibers/.nexus/nexus.db` (rig rows live in `books_spider_rigs` as JSON blobs). Confirmed `RigStatus` / `EngineStatus` enums via `packages/plugins/spider/src/types.ts`.
- Backed up the DB to `/tmp/nexus.db.bak-1776133100`.
- **First patch attempt:** reset rig → `running`, `inventory-check` + five cancelled downstream engines → `pending`. Write committed cleanly; Spider immediately re-ran `inventory-check`, hit the same empty plan, and snapped the whole rig back to `failed`.
- **Diagnosis of why:** inspected reader session `ses-mnxy3rbw-fa6db8ac`. Session exited 0 / status `completed`, but the `output` field is the model explaining it *couldn't* call `inventory-write` because the guild daemon (PID 32962) was stale and every MCP call returned HTTP 500. The reader dumped its inventory into prose in the final message instead. `books_astrolabe_plans` row for the writ has no `inventory` key at all.
- **Second patch (option 1 — re-run reader):** reset `reader` as well, plus the tail, rig → `running`. Patch stuck. Verified via `nsg rig show`: rig `running`, reader immediately picked up and transitioned to `running`, everything downstream waiting in `pending`.

## Decisions made and/or deferred

- **Chose option 1** (re-run reader with live daemon) over hand-authoring an inventory from the reader's prose output, re-commissioning, or fixing the underlying bug first. Rationale: cleanest path that still produces a tool-written inventory for the experiment record, at the cost of ~7 min of reader session time.
- **Deferred:** the underlying bug. A reader session that couldn't write its inventory but still exited 0 and reported `completed` is a real gap — either the session should fail, or `inventory-check` should run as a guard inside the reader engine, not as a separate downstream engine that cancels everything on miss. No quest opened yet; worth one.
