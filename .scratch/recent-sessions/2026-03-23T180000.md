# Session Summary

## What we did

- **Designed the guild system architecture** from scratch across a long collaborative session. Produced two documents:
  - `.scratch/guild-infrastructure-plan.md` — the universal guild system architecture (framework-agnostic, any guild could use this)
  - `.scratch/nexus-guild-plan.md` — Sean's specific guild organization and migration plan
- **Named Nexus as the framework** — the runtime that makes guilds operational, with CLI commands (`nexus init`, `nexus install`, `nexus repair`) and base implements/machines
- **Defined the five pillars**: commission pipeline, stores, instruction composer, register, and the pulse (future)
- **Established the full HQ structure**: codex, training (curricula + temperaments), stores (implements + machines), nexus/ (framework-managed base tools), guild.json as central index, Ledger (SQLite) for operational state
- **Designed anima composition**: curriculum + temperament + oaths, assembled by summon machine via template, no per-anima markdown files
- **Split framework from guild tools**: `nexus/` (framework territory, repairable) vs `stores/` (guild territory, leadership-owned) — physical separation for safety

## Decisions made and/or deferred

- **Codex** = institutional policy/procedures; **Implements** = CLI tools animas wield; **Machines** = automated processes; **Relic** = load-bearing legacy artifact (current `nx` CLI)
- **Temperament** chosen as the name for personality templates; produced by the academy alongside curricula
- **Golems renamed to machines** — "too pretentious for a bash script"
- **Bare mode, no CLAUDE.md** — animas get all instructions via composed system prompt, not repo config
- **guild.json as single index** for active tools, workshops, curricula, temperaments (not the Ledger)
- **Ledger schema owned by framework**, migrations in `nexus/migrations/`
- **Forge and academy are guild-specific organizational choices**, not framework requirements
- **Deferred**: dispatch/summon boundary detail, machine authorship governance, commons/credentials, performance assessment

## Next steps & open questions

- Populate HQ: extract codex from existing teachings, author initial curricula/temperaments
- Create forge and academy workshops

### Notable moments

- **"The guild itself, including its internal details, is its first work"** — Sean's renewed philosophy driving the entire session. He came in ready to dictate technical architecture after reconciling the philosophical tension between hands-off patron and hands-on founder.
- **Forge and academy are not foundational** — Sean realized mid-session that these are his organizational preferences, not system requirements: "another guild master could create all those artifacts by hand." This split the architecture into universal framework vs. specific guild, which led directly to Nexus as a named framework.
- **The repair/isolation insight** — Sean wanted to protect base tools from guild tinkering. This crystallized the `nexus/` vs `stores/` physical separation and the `nexus repair` concept — the framework as a "system partition" that can be restored independently.
