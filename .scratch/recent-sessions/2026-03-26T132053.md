## Decisions made and/or deferred

- **Stateless turns over stateful objects.** Sean pushed for DB-backed state instead of in-memory `Conversation` class. Any component can drive a turn with just a conversation ID. Process restarts don't kill conversations.
- **Session funnel reuse over bypass.** Each turn is a `launchSession()` call — no parallel path. Added `claudeSessionId` to thread `--resume` through the existing infrastructure.
- **Manifest at turn time.** Animas are manifested fresh each turn, not at conversation creation. Picks up guild changes mid-conversation.
- **Turns are sessions.** No separate turns table. Each turn produces a session row with full metrics. Conversation tables are thin grouping/state layer.
- **No denormalized aggregates.** Cost/token totals computed via `SUM() GROUP BY conversation_id`. SQLite is fast enough.
- **Streaming: Option A (launchStreaming on provider interface)** over callback approach. Cleaner contract, optional method.
- **Full tool access for convened animas.** Per Sean: "that's the point." Trust role gating.
- **Budget caps deferred.** Session funnel can hook into generalized capping later.
- **`--resume` and manifest:** Will manifest every turn (Sean's direction), not try to optimize by caching across turns.

### Notable moments

- **"is there a way we store conversations in a database... no references have to be held, just the relevant conversation ID"** — Sean rejected the stateful `Conversation` class with generators in favor of pure DB-backed state. This was the pivotal design shift — turned a complex OOP design into a simple stateless function. **Notable because:** Sean identified that process resilience and component independence mattered more than API ergonomics.

- **"is there a reason we don't manifest anima the standard way?"** — Coco had built a parallel path where callers prepared session files and passed paths to the Conversation constructor. Sean's question collapsed the entire `prepareSessionFiles` / path-passing machinery. **Notable because:** the simplest question eliminated the most code.

- **"similarly... are there issues with the session funnel that makes us unable to use it"** — Sean pushed back on bypassing `launchSession()`. Coco had assumed incompatibility; Sean asked to verify. Turns out the only real change needed was one field (`claudeSessionId`). The session funnel handled everything else. **Notable because:** three rounds of "does this actually need to be different?" — stateful→stateless, custom manifest→standard manifest, bypass funnel→use funnel — each one simplified the design dramatically.

- **"why is there no new tool to convene a conversation?"** — After implementation was "done," Sean spotted the missing piece: animas themselves should be able to initiate a convene via tool call. Led to the `convene` MCP tool.
