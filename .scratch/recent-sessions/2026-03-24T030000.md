# Session Summary

## What we did

- **Added CI workflow** (`.github/workflows/ci.yml`) — build, typecheck, test on PRs and main pushes.
- **Added publish workflow** (`.github/workflows/publish.yml`) — publishes all 12 `@shardworks/*` packages on `v*` tag push. Supports both `NPM_TOKEN` secret (for bootstrapping) and OIDC trusted publishers (steady state).
- **Added release script** (`scripts/release.sh`) — bumps unified version across all 13 package.json files, commits, tags. User pushes to trigger publish.
- **Prepared all packages for npm publishing** — added `publishConfig.exports` (mapping `.ts` source to `dist/` output), `files: ["dist"]`, excluded test files from tsc build. Zero impact on local dev workflow.
- **Added canonical version** (`0.1.0`) to root `package.json`.
- Committed as `8526cb1` and `7c4aa5b`, pushed to `origin/main`.

## Decisions made and/or deferred

- **Unified versioning** — all packages share one version number. Simpler mental model for a tightly-coupled framework at 0.x. Can migrate to changesets later if needed.
- **No Lerna** — Sean: "lerna feels annoying for this." Plain bash release script instead.
- **tsc build, not bundling** — `publishConfig` swaps exports at publish time so local dev stays on raw `.ts` with Node 24 transform-types.
- **Dual-mode auth** — `NPM_TOKEN` for bootstrap, OIDC trusted publishers for steady state. Sean's idea to avoid the first-publish chicken-and-egg problem.
- **Deferred: guild tool dependency resolution** — guild-installed implements that import third-party packages will fail at runtime because `install-tool` skips `node_modules`. Framework tools are fine (resolved from npm). Design problem captured in conversation, not yet filed.
