## Context

Sean kicked off this session wanting to figure out the best way to capture project fundamentals — the kind of foundational knowledge that agents need to do their work and that persists across sessions. The project is very early: there's a devcontainer setup, a nearly-empty `CLAUDE.md`, and some initial memory files, but no application code or formal project documentation yet.

This is a design question, not an implementation question: *where should foundational project knowledge live, and in what form?*

## Current State of Knowledge Capture

What exists today:

- **`CLAUDE.md`** — Nearly empty. Intended for actionable agent instructions.
- **Memory files** (`~/.claude/projects/.../memory/`) — Two files capturing Sean's preferences and high-level project context. These persist across conversations but live outside the repo.
- **Session docs** (`docs/sessions/`) — This is the first one. Narrative records of human-agent interactions.

## Key Decision: Two Human-Owned Foundational Artifacts

Sean established that two artifacts are under his **plenary control** — baseline requirements for the entire project. Agents work within these, never modify them directly.

### 1. Requirements (Functional & Non-Functional)

The objectives the entire system is constructed to meet. Includes:
- What the system must do
- What it must never violate
- How it must perform

These are not negotiable by agents. They are the acceptance criteria for everything.

### 2. Vocabulary — A Typed Ontology

A shared conceptual language for both humans and agents, with two layers:
- **Prose definitions** — human-readable, making requirements expressible and discussable
- **Formal typed ontology** — every named domain concept gets a formal definition, exported by a top-level module

Key design decisions:
- **Statically verifiable source of truth.** The formal types are the authority, not the prose.
- **Compliance is a hard gate.** Agent contributions must conform to the domain ontology.
- **Implementation behind the interfaces is opaque.** Sean owns the surface (what things are called, what they mean, what they must do). Agents own everything behind the interface.
- **Sean envisions a "system REPL"** — the possibility of an operator interacting directly with domain entities through a live interface.

### Mutability Process

- Agents may *propose* changes to requirements or vocabulary (e.g., "I think we need a new domain concept").
- Sean approves or rejects. The artifacts themselves are never directly modified by agents.

## Decision: Structural Protection via Separate Repo + Read-Only Mount

The core threat model: agents running with full permissions could modify requirements or ontology files to "meet requirements" rather than actually meeting them. Instruction-based protection (CLAUDE.md, pre-commit hooks) is a soft gate — agents can bypass it.

### Options Considered

1. **Read-only volume mount** — Domain directory mounted `:ro` in containers. Writes physically fail. ✅ Selected.
2. **Git submodule** — Standard mechanism but fiddly, and agents could theoretically modify submodule content.
3. **Separate repo, manual path reference** — Complete separation but feels disconnected.
4. **Pre-commit hook rejection** — Soft gate, doesn't solve the threat model.

### Outcome

- Created **`shardworks/nexus-mk2-domain`** as a public GitHub repo (public because the experiment is open).
- Domain repo is cloned as a sibling to the main repo (`~/sandbox/nexus-mk2-domain`).
- A host-side initialize script (`20-clone-domain-repo.sh`) ensures the domain repo is present and up to date before the container starts.
- Mounted read-only at `/workspace/domain` inside containers via `devcontainer.json`.
- `CLAUDE.md` updated with a pointer to `/workspace/domain/` and explicit "do not modify" instructions.

### Initial Structure

```
nexus-mk2-domain/
  README.md
  requirements/
    index.md          # All requirements (single doc for now, granularity evolves later)
  ontology/
    index.ts          # Single export surface for all domain types
```

### Design Principles Captured

- `CLAUDE.md` stays lean — actionable instructions and pointers, managing token burn.
- Domain files are physically read-only for agents — not just instructionally.
- Requirements start as a single document; granularity evolves organically.
- The ontology is TypeScript — statically verifiable, good tooling, consumable at runtime.
- Implementation behind the ontology interfaces is opaque — agents own that.
